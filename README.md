# Hunter —— 消费级具身 AI 代理，第一台为不会说话的存在而生的机器

> "这不是给猫的玩具。这是给猫的同事。"

---

## 它从一个让人难受的事实开始

你的猫，每天有九个小时是独自度过的。

你上班，它在家。沙发、阳光、窗台、沉默。它天生是猎手，每一根神经都为追逐和伏击而设计，但现代公寓把它变成了一个没有猎物的猎人。

市面上不是没有解决方案。自动逗猫棒、随机转动的激光、定时弹出的羽毛——它们的设计逻辑都一样：给猫一个规律的刺激，让它不那么无聊。

问题在于，猫不傻。**三天。** 任何一个可以被预测的运动模式，猫三天就识破，然后永远失去兴趣。

传统玩具给的是"刺激"，但猫需要的是"博弈"。它需要一个真正的猎物——一个会逃跑、会反击、永远无法被完全预测的对手。

Hunter 就是这个对手。

---

## Hunter 是什么

一句话：**一个能跑、能看、能想、能逗、能拍、能讲故事的桌面级具身 AI 代理——专为独居家庭的猫设计，以"猎手"为人格。**

但这个定义远不够准确。

家庭 AI 产品分两种：被动工具（音箱、扫地机）和拟态宠物（Lovot、Ropet）。Hunter 都不是。它是**有自己目标的代理**——它的目标是"让这只猫过得更好"，这个目标是它自己持续在追求的，不是你按一下按钮才执行。

传统的家庭关系结构是：**主人 ↔ 猫**。Hunter 把它改写成：**主人 ↔ Hunter ↔ 猫**——一个新进入这个家的第三方，会和主人协作，也会和猫建立独立关系。

这不是宠物用品赛道。这是**消费级具身 AI 代理的第一台量产机**。

## 三个让这个项目变得不一样的洞察

### 洞察一：你卖的不是"陪伴"，是"你不在的那九小时"

所有宠物用品本质上都在做"主人在场时锦上添花"——猫树、零食、漂亮玩具。Hunter 是唯一一个专门为"主人不在"设计的产品。

它不和猫树竞争，猫树服务的是"在场"；它不和真人陪伴竞争，人陪不了九小时；它甚至不和其他智能玩具竞争，那些是"零食式"产品，不是"代理式"。

主人付 1999 元买的不是一个机器人，是 **"我不在的时候，我的爱依然在场"** 这个心理状态。

### 洞察二：AI 不该追求最优，它必须主动制造意外

ChatGPT 追求"最准确"，Tesla FSD 追求"最高效"，推荐算法追求"最点击"——它们都在优化。但 Hunter 不能优化。**优化是猫无聊的根源。**

Hunter 引入了一个全新的算法指标：**"惊喜熵"（Surprise Entropy）**——实时衡量当前动作模式对这只猫的不可预测程度。算法目标不是"让猫扑得更准"，而是"让猫始终保持 70% 的把握和 30% 的不确定"。

这是 Csikszentmihalyi 心流理论在物理交互上的工程实现。把游戏设计理论、即兴喜剧原理和具身 AI 融合在一起——相关文献几乎空白。

Hunter 会故意失败，让猫扑中；会故意打破规律，在猫预测下一步前突然反转；会故意"笨拙"，模拟受伤的猎物——这是真实捕食学里的诱饵机制。当它发现某个套路猫开始熟练，会立刻主动弃用。

### 洞察三：这是第一个为"不会说话的存在"设计的 AI

现有所有 AI 产品——ChatGPT、Siri、Copilot——都依赖**语言**作为接口。但世界上有大量主体是不会说话的：婴儿、宠物、失语老人、自闭症患者……

Hunter 在做的事，本质是用 AI 解读非语言主体的需求。设备读的是：尾巴姿态（垂直警戒还是水平放松）、耳朵角度（前倾专注还是后压恐惧）、瞳孔大小（兴奋还是应激）、全身姿态（潜伏还是蜷缩）。

这是 ambient computing 的极致——AI 在没有任何主动指令的情况下读懂主体。

它的英文定义可能是：**"AI for the Inarticulate."**

今天做猫，明天的技术栈可以平移到老年宠物、失能老人、低龄婴儿——伦理光环加成，市场无限大。

## 它能做什么

### 第一步：读懂这只猫

在做任何事之前，Hunter 先花时间理解它面对的是谁。

广角主摄持续扫描房间，**视觉追踪**锁定猫的实时位置和姿态——尾巴垂直还是水平、耳朵前倾还是后压、身体潜伏还是蜷缩——每一帧都在解读情绪。麦克风同步进行**叫声识别**，把不同音调的喵声分类为饥饿、撒娇、警戒、玩耍，翻译成主人能理解的意图标签。结合运动轨迹和互动频率，Hunter 持续计算**活跃度**，判断这只猫今天是精力充沛还是需要安静。

时间久了，Hunter 会建立起这只猫独有的**个性档案**：它更偏爱羽毛还是激光，喜欢黄昏还是清晨，对快速逼近是兴奋还是警惕。这些是主人自己也未必说得清楚的偏好。

### 第二步：开始一场真正的狩猎

读懂状态之后，才是互动的开始。

Hunter 自主巡航房间，在合适的时机展开机械臂。**逗猫棒**以生成式动作模拟猎物——不是预设序列，每一次都是实时计算的新轨迹，永远无法被预测。猫扑空时"猎物逃脱"再启动，扑中时"被捕获"。需要换个玩法时，切换到**激光模式**，在地板和墙壁之间制造光点游走，诱导猫进行全速冲刺追击。

扑中目标的瞬间，底盘投出一颗零食。

**狩猎 → 捕获 → 进食**——这是完整的生物学闭环，也是任何现有产品都没有做到的事。

### 第三步：悄悄照料

互动之外，Hunter 还在做一些猫感知不到、但主人需要知道的事。

摄像头每隔一段时间抓取快照，AI 分析猫靠近水碗食碗的频次和时长——**饮水和饮食异常**是宠物各类疾病的早期信号，也是肉眼最难发现的一种。当活动量或步态出现连续异常，系统会在症状明显之前向主人发出预警，把被动等待变成主动干预。

猫独处时偶尔需要安抚。Hunter 会播放预录的猫咪音效（`cat1`–`cat4`）吸引注意，或者在主人录制语音后定时播放，让猫在家里听到熟悉的声音——不只是玩伴，也是陪伴。

### 第四步：把一切讲给你听

下班路上，手机推送当天最佳 30 秒高光视频。推门前你已经知道它今天过得怎么样。

每晚的**AI 日记**不是数据图表，是 LLM 写的文字：

> "今天 Mochi 在阳光下睡了两小时，醒来后试图捕捉那只羽毛玩具 23 次，差两次几乎得手，失手后假装根本不在乎。黄昏时分状态最好，爆发了三次完整的冲刺追击。今天的活力指数：92。"

AI 还会把最有表情的瞬间自动裁剪成**表情包**，直接存进手机相册。出门在外想它了，打开 App **远程接管**，用第一视角操控 Hunter 实时陪它玩——异地云吸猫，不是比喻。

### 记忆盒子（Memory Box）

这是一个需要慎重处理，但一旦理解就难以拒绝的功能。

猫的平均寿命 12 到 18 年。Hunter 在猫的整个生命中持续记录每一个互动瞬间。当猫离世后，Hunter 自动整理出"它的一生"——从第一次扑跳的笨拙、到中年的鼎盛、到老年的优雅——配 AI 写的回忆录、最佳瞬间合集、最有它"性格"的 100 个 30 秒视频。

营销时不写"如果你的猫死了"，只写：

**"Hunter 会记得它生命的每一天。"**

### 每台 Hunter 最终都不一样

出厂时所有设备相同。六个月后，张三家的 Hunter 学会了张三家猫不喜欢被快速逼近、喜欢黄昏运动、对羽毛比对鼠形敏感；李四家的 Hunter 学的是另一套。

这意味着：**用户不能换设备**。如果你想换一台新机器人，你失去的是六个月的人格化记忆。这是真正的 SaaS 式锁定，比订阅更深。

## 系统架构

五个功能模块通过统一的 REST API 调用硬件，上层逻辑与硬件完全解耦。

```
┌──────────────────────────────────────────────────────────┐
│                     五个功能模块                          │
│                                                          │
│  perception/      hunt/       care/    report/  memory/  │
│  ─────────────   ──────────  ───────  ───────  ───────  │
│  视觉追踪         逗猫棒       饮水      AI日记   记忆盒子  │
│  叫声识别         激光追逐     健康      表情包             │
│  活跃度感知       零食闭环     安抚      远程接管           │
│  个性档案                                                 │
└────────────────────────┬─────────────────────────────────┘
                         │  HTTP REST  (api_client.py)
┌────────────────────────▼─────────────────────────────────┐
│            Robot Control Unit  (FastAPI)                  │
│                  http://192.168.0.170:8000                │
│                                                          │
│  /api/cmd/{action}    移动·避障·音效·速度                 │
│  /api/camera/         摄像头流 · 单帧快照                  │
│  /api/audio/          录音 · 播放                         │
│  /api/state           状态 · 日志                         │
│  [ /api/arm/  →  机械臂  ]   [ /api/yolo/ →  检测  ]     │
└────────────────────────┬─────────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────────┐
│                      硬件层                               │
│   USB 摄像头   底盘·麦克纳姆轮   避障超声波   音频系统      │
│   [ 机械臂 4-5 DoF → ]   [ 投食机构 → ]                  │
└──────────────────────────────────────────────────────────┘
```

**代码目录结构：**

```
src/
├── api_client.py        # 统一封装所有硬件 API 调用
├── perception/          # 第一步：感知
├── hunt/                # 第二步：狩猎
├── care/                # 第三步：照料
├── report/              # 第四步：汇报
│   └── meme_generator.py
└── memory/              # 记忆盒子
```

当前已封装的指令集：

| 类别 | 指令 |
|------|------|
| 移动 | `forward` `backward` `left` `right` `rotate_cw` `rotate_ccw` `stop` |
| 安全 | `obstacle_on/off` `ultra_on/off` `emergency` |
| 速度 | `speed_up` `speed_down` `speed_reset` |
| 音效 | `cat1` `cat2` `cat3` `cat4` |
| 摄像头 | `/stream.mjpg` `/snapshot.jpg` |
| 录音 | `rec/start` `rec/stop` + 播放 |

机械臂、YOLO 目标检测、投食机构将以同样方式扩展进该 API，上层模块无需改动。

## 为什么 V1 只做一只猫

25–35 岁，一线城市，独居，有一只 1–8 岁活力健康猫，工作日 9 小时以上不在家。

这是 V1 的唯一目标用户。

我们不做多猫家庭（V2 再做），不做老年猫（安全要求不同），不做病猫（医疗器械范畴），不做狗——狗的运动尺度和猫差异太大，做不好两边都伤。

这种"敢说不"，比"什么都支持"更有力量。

## 当前进展：48 小时 MVP

赛场上宁可展示一个完美运行的单一功能，也不展示三个会卡顿的半成品。

MVP 保留两个核心：

**反射弧**：轻量 YOLOv8 nano 实时识别猫爪位置，状态机驱动机械臂——目标靠近时瞬间闪避，目标静止后主动试探。评委伸手去抓，机械臂躲开。这是最具视觉冲击力的具身智能演示。

**运动仪表盘**：视觉捕捉猫的移动轨迹，实时生成热力图和统计数据（互动时长、最高移动速度、爆发冲刺次数）。这证明它不只是执行器，更是数据采集终端。

底层运行在树莓派 5 上，这份代码仓库包含摄像头流传输和远程监控的基础实现。

## 更大的图景

Hunter 的本质不是宠物产品，是一套**读懂非语言主体需求的 AI 范式**。

猫是第一个用例，因为它足够明确、足够高频、足够有情感浓度。但这套技术——用体态识别替代语言指令，用反向数据流替代主动交互——可以迁移到任何不会说话的存在：特殊需求儿童、失语老人、新生儿监护。

今天是消费级宠物 AI。这条路的终点，是 **AI 与不会说话的世界之间的翻译器**。
